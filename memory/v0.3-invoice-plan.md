# v0.3 — Invoice & Charge Management: Implementation Plan

> Branch: `feature/invoice-management`
> Replaces the `New Debt` / `Paid Debt` transaction hack with a proper receivables system.

---

## Current System — What We're Replacing

**How debt works today:**
- Cashier adds `New Debt` as a line item in POS (hardcoded item name, not a catalog service)
- Stored as a regular `transaction` document: `{ item: 'New Debt', price: amount, customerName, customerId, shiftId }`
- Debt payment is `Paid Debt` — another transaction with a negative-ish effect
- `DebtReport.jsx` reads all transactions, client-side filters for `item === 'New Debt' || item === 'Paid Debt'`, and computes running balance per customer
- No status, no partial payment tracking, no connection to an actual order
- `shiftFinancials.js` treats `New Debt` as an expense (`EXPENSE_ITEMS` set)
- Forward-proofing already exists: `invoiceStatus: 'UNPAID'` field referenced in `DebtReport` filter

**What exists that we can build on:**
- `customers` collection — `{ username, fullName, address, tin, createdBy, createdAt }`
- `CustomerDialog.jsx` — search + create customers, already used in POS
- `ServiceInvoice.jsx` — printable invoice component (used for order receipts, not AR — we extend this)
- `OrderManagement.jsx` — order detail views; invoice action can live here too

## Data Model

### New: `invoices` collection

```
invoices/{id}
  // Source
  orderId: string                  // order that created this invoice (null if manually created)
  orderNumber: string              // display ref

  // Customer
  customerId: string
  customerName: string
  customerAddress: string          // snapshot at invoice time
  customerTin: string              // snapshot at invoice time

  // Financials
  items: [                         // snapshot of order items at invoice time
    { description, quantity, price, total }
  ]
  subtotal: number
  total: number
  amountPaid: number               // running total of all payments received
  balance: number                  // total - amountPaid (denormalised, updated on payment)

  // Status
  status: 'unpaid' | 'partial' | 'paid' | 'written_off'
  // 'overdue' is a derived display state (balance > 0 && dueDate < now) — not stored
  dueDate: timestamp               // required; set at creation, pre-filled from settings/config → invoiceDueDays

  // Metadata
  notes: string
  createdAt: timestamp
  staffEmail: string               // who created it
  shiftId: string                  // shift it was created in

  // Payments log
  payments: [
    {
      paymentId: string            // uuid
      amount: number
      method: 'cash' | 'gcash'
      date: timestamp
      staffEmail: string
      note: string
    }
  ]
```

**Status transitions:**
```
unpaid → partial (first partial payment recorded)
partial → paid   (balance reaches 0)
unpaid  → paid   (full payment recorded)
any     → overdue (batch/scheduled check: balance > 0 && dueDate < now)
```

**Firestore indexes needed:**
- `invoices` — `customerId ASC, createdAt DESC`
- `invoices` — `status ASC, createdAt DESC`
- `invoices` — `staffEmail ASC, createdAt DESC`
- `invoices` — `shiftId ASC` (for shift-level AR summary)

## Touchpoints — What Changes Where

### 1. POS Checkout (`CheckoutDialog.jsx`)

**New payment method option: "Charge to Account"**
- Appears alongside Cash / GCash in the payment method ToggleButtonGroup
- Selecting it requires a customer to be assigned to the order (existing customer picker enforces this already)
- On confirm: instead of closing cleanly, calls `createInvoice()` — writes to `invoices` collection, marks the order with `invoiceId` and `paymentMethod: 'charge'`
- No change to how `orders` or `transactions` are written (they still record the sale); only the receivable is created separately
- `New Debt` removed from POS item list once this is live

**Remove from POS:**
- `{ id: 'pd', serviceName: 'Paid Debt' }` from the services combo list
- `'New Debt'` hardcoded item handling in `handleAddToOrder()`
- Debt customer picker tied to `New Debt` / `Paid Debt` item logic

### 2. New Admin Screen: `InvoiceManagement.jsx`

Route: `/admin/invoices`

**List view:**
- Table: Invoice #, Customer, Date, Due Date, Total, Paid, Balance, Status chip
- Filter chips: All / Unpaid / Partial / Paid / Overdue
- Search by customer name or invoice number
- Click row → detail panel or dialog

**Detail panel / `InvoiceDetailDialog.jsx`:**
- Full invoice summary (items, totals)
- Payment history log
- "Record Payment" button → opens `RecordPaymentDialog`
- "Print Invoice" button → triggers existing `ServiceInvoice` print component (extended)
- Status chip (editable: can manually mark overdue / write off)

**`RecordPaymentDialog.jsx`:**
- Amount field (pre-fills with current balance for full payment)
- Method: Cash / GCash
- Optional note
- On save: appends to `payments[]`, updates `amountPaid`, `balance`, recalculates `status`
- Uses Firestore `updateDoc` with `arrayUnion` for the payments array

**Write-off action** (in `InvoiceDetailDialog`, separate from Record Payment):
- Triggers `ConfirmationReasonDialog` — reason required
- Appends `{ method: 'write_off', amount: balance, note: reason, ... }` to `payments[]`
- Sets `status: 'written_off'`, `balance: 0`
- Excluded from Outstanding Receivables total and overdue display

### 3. Replace `DebtReport.jsx` → `ReceivablesReport.jsx`

- Reads from `invoices` collection (not `transactions`)
- Groups by customer, shows total outstanding
- Preserves old `New Debt` / `Paid Debt` transactions in a "Legacy" section (read-only, date-filtered)
- Links customer row → `AdminDebtLookupDialog` (repurposed to show invoice history)

### 4. Customer Profile: Invoice History

- `AdminDebtLookupDialog.jsx` extended to show invoices tab alongside legacy debt transactions
- Outstanding balance computed from `invoices` (not running sum of `New Debt` transactions)

### 5. Dashboard Widget (`AdminHome.jsx` / `KpiRow.jsx`)

- New KPI card: **Outstanding Receivables** — sum of `balance` across all `status !== 'paid'` invoices
- Firestore: `collection('invoices'), where('status', '!=', 'paid')` — client sum of `balance`

### 6. Shift Financials (`shiftFinancials.js`)

- Remove `New Debt` from `EXPENSE_ITEMS` set (once migration period ends)
- New Debt transactions logged during the transition period remain classified as expenses (backward compatible)
- Future: `invoices` created in a shift can be summarised separately if needed (out of scope for v0.3)

### 7. `ServiceInvoice.jsx` — extend for AR invoices

Currently renders order receipts. Extend to also support AR invoice format:
- Show `Due Date` and `Balance Due` sections
- Show payment history if partially paid
- Keep existing order-receipt mode (prop-driven: `mode: 'receipt' | 'invoice'`)

## Migration / Backward Compatibility

- **Old `New Debt` / `Paid Debt` transactions are never deleted** — `DebtReport` (or its replacement) keeps a legacy view for them
- **No data migration required** — old system runs in parallel until manually wound down
- **Cutover**: once `InvoiceManagement` is live, the `New Debt` / `Paid Debt` items are removed from POS. Existing balances tracked in old system remain visible in the legacy section of `ReceivablesReport`
- **`invoiceStatus: 'UNPAID'`** — the forward-proofing field already in `DebtReport`'s filter can be retired once the new system is the source of truth

## Decisions

- **Due date** — Set per invoice at creation time. Cashier picks the date in the "Charge to Account" flow. Admin can set a default offset (e.g., 7 days) in `settings/config → invoiceDueDays`; the date field pre-fills from that default but is always editable before saving.

- **Write-off** — Supported. Status adds `written_off`. A written-off invoice shows the original balance in the ledger but is excluded from the Outstanding Receivables total and overdue counts. Requires a reason (uses existing `ConfirmationReasonDialog`). Stored as a payment-log entry with `method: 'write_off'` and the reason in `note`.
  - Status set: `invoices` — `status: 'unpaid' | 'partial' | 'paid' | 'overdue' | 'written_off'`

- **Overdue detection** — Client-side at render time: compare `dueDate` to current timestamp. Any invoice with `balance > 0 && dueDate < now` displays as overdue. No Cloud Function needed for v0.3. `status` field in Firestore is only written on payment events (not auto-updated to `overdue`); overdue is a derived display state, not a stored status change — avoids needing a scheduled function and keeps the data model simpler.

- **GCash charge accounts** — Payment method on the invoice (`RecordPaymentDialog`) is the settlement method. The charge-to-account action itself is payment-method-agnostic. Customer can owe ₱500 and later pay it via Cash, GCash, or a mix across multiple partial payments.
